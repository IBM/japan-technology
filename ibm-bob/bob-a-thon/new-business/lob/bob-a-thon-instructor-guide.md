## 📋 このガイドについて

Bob-a-thonハンズオン・ワークショップを運営する講師向けのガイド。事前準備、当日の進行、ファシリテーションのポイント、トラブル対応をまとめている。

**想定する講師**: IBM Bobの操作経験が浅い方（初めて〜少し触った程度）でも運営できるように書かれている。

**対応する受講者ガイド**: [こちら](https://github.com/IBM/japan-technology/blob/main/ibm-bob/bob-a-thon/new-business/lob/readme.md)
 を参照しながら読むこと。

## 1. 事前準備

### 1週間前: 環境確認

以下をチェックし、問題があれば早めに対処する。

#### 講師側の確認

- [ ] IBM Bobが自分のPCで起動・動作するか確認済み
- [ ] IBM Bobのチャット欄でメッセージを送信し、応答が返ることを確認済み
- [ ] `/ask`、`/code`、`/plan` コマンドが動作することを確認済み
- [ ] Webブラウザー（Chrome推奨）でHTMLファイルを開けることを確認済み
- [ ] 配布用のlab-scenarioフォルダの内容を確認済み（[配布フォルダの準備](#配布フォルダの準備)を参照）

#### 参加者側の確認

- [ ] 全員のPCにIBM Bobがインストールされているか確認済み
- [ ] IBM Bobのライセンス・アカウント設定が完了しているか確認済み
- [ ] ネットワーク環境（IBM Bobがサーバーと通信できるか）を確認済み

### 配布フォルダの準備

`lab-scenario/` フォルダの中身を確認し、必要に応じてカスタマイズする。

```
lab-scenario/
├── company_profile.md              企業設定メモ（空欄テンプレート）
├── sales_challenge.md              営業チームの課題カード
├── marketing_challenge.md          マーケティングチームの課題カード
├── portal_challenge.md             顧客管理チームの課題カード
├── onboarding_challenge.md         人事チームの課題カード
└── mockup-req-template-sample.md   要件テンプレートのサンプル
```

**カスタマイズのポイント:**

- 参加者の業界が決まっている場合は、課題カード内の「業界別の適用例」を該当業界のものに絞ると、よりリアルな演習になる
- `mockup-req-template-sample.md` は Lab 2 でBobが生成するテンプレートの参考例。事前に自分で `/ask` を実行して生成した結果に差し替えてもよい
- `company_profile.md` は空欄のまま配布する（受講者がチーム演習で記入する）

**ZIPファイルの作成:**

1. `lab-scenario` フォルダを右クリック → 「圧縮」（macOS）または「送る → 圧縮(zip)フォルダー」（Windows）
2. 生成された `lab-scenario.zip` を参加者に配布する

### 講師リハーサル（前日までに実施）

**所要時間の目安: 60〜90分**

受講者ガイド（v2.6）のLab 1〜6を、受講者と同じ手順でひと通り実施する。以下を確認する。

1. **Lab 1**: ざっくり依頼で生成されるCRMモックアップの内容を把握する。Bobの応答時間を計測しておく
2. **Lab 2**: `/ask` で生成される要件テンプレートの中身を確認する。`mockup-req-template-sample.md` と大きく異なる場合は、サンプルの差し替えを検討する
3. **Lab 3**: `@ファイル名` のコンテキスト・メンションが正しく動作するか確認する。[Add File] メニューが表示されるか確認する
4. **Lab 4**: `/code` で生成されるモックアップの品質を確認する。HTMLファイルがブラウザーで正しく表示されるか確認する
5. **Lab 5**: ダーク・テーマへの変更が反映されるか、`crm_req.md` が更新されるか確認する
6. **Lab 6**: 設計書と説明書が生成されることを確認する

> 💡 リハーサルで生成したモックアップは、当日のデモで見せる際の参考にもなります。スクリーン・ショットを撮っておくとよいでしょう。

## 3. ファシリテーション・ノート

### 導入（0:00〜0:03）

**伝えること:**

- このワークショップの目的: IBM Bobを使って「要件 → モックアップ → 文書化」の流れを体験する
- 2部構成: 前半の「練習ラボ」で全員同じ題材（CRM）を使って基本を学び、後半の「チーム演習」で自分たちの課題に取り組む
- ギーク思考の4規範を紹介する（受講者ガイドのテーブルを見せる）。深く説明せず「各Labで体感してもらいます」と伝える

**話し方の例:**

「今日はIBM Bobというツールを使って、モックアップを作る体験をします。ただ作るだけではなく、要件をまとめてから作る、作った後にドキュメントも揃える、という仕様駆動のプロセスを体験します。前半は全員で同じ課題に取り組み、後半はチームに分かれて自分たちの課題に挑戦します。」

### Lab 1: ざっくり依頼（0:03〜0:08）

**冒頭で伝えること:**

「まずは細かいことは考えずに、Bobにざっくり頼んでみます。どんなものが出てくるか、試してみましょう。」

**講師のアクション:**

1. 受講者ガイドの手順1-1〜1-2をデモで見せながら、一緒に進める
2. 手順1-3のプロンプトをコピペで実行するよう案内する
3. Bobが応答している間に、「出来上がったらブラウザーで開いてみてください」と案内する

**Bobの応答待ち中に話すこと:**

「Bobが作っている間に少し補足します。今、具体的な指示を出さずに頼みましたよね。出来上がったものを見て、『自分が思っていたものと違う』『これが足りない』と感じるポイントがあれば、覚えておいてください。次のLabで、その差を埋める方法を学びます。」

**よくある質問:**

| 質問 | 回答 |
|---|---|
| 「チャット欄はどこですか？」 | IBM Bobの画面にあるチャット・パネルを指す。位置はレイアウト設定によって異なる |
| 「フォルダの開き方がわからない」 | 「ファイル」→「フォルダーを開く」の手順を画面で見せる |
| 「HTMLをブラウザーで開けない」 | HTMLファイルを右クリック →「エクスプローラーで表示する」/「Finderで表示する」→ ファイルをダブルクリック |

**冒頭で伝えること:**

「Lab 1ではざっくり頼みました。ここからは、もう少し丁寧に依頼するやり方を学びます。まずは、要件を整理するための『型』をBobに作ってもらいます。」

**講師のアクション:**

1. `/ask` コマンドの意味を簡単に説明する（「Askモード — コードは書かず、調査やドキュメント作成を頼むモード」）

**Bobの応答待ち中に話すこと:**

「出来上がったテンプレートは、次のLabで要件を書くときの『型』として使います。テンプレートの中身を見て、『事業担当者が書ける項目が前半に、技術的な項目が後半に』という構成になっているか確認してみてください。」

**よくある質問:**

| 質問 | 回答 |
|---|---|
| 「`/ask` と普通のチャットの違いは？」 | `/ask` はAskモードに切り替わり、コードを書かずに調査やドキュメント作成に特化する。普通のチャットでも似たことができるが、モードを明示すると意図が伝わりやすい |

## 4. よくあるトラブルと対処法

| トラブル | 原因 | 対処法 |
|---|---|---|
| Bobが応答しない | ネットワークの問題、またはサーバー側の負荷 | ネットワーク接続を確認する。数分待ってから再送信する。それでもダメなら、IBM Bobを再起動する |
| Bobの応答が非常に遅い | 複数の参加者が同時にリクエストしている | 「少し時間がかかることがあります」と伝え、応答待ちの間に補足説明を行う |
| 生成されたHTMLが正しく動作しない | Bobの生成コードにバグがある | Bobに「ブラウザーで開くと〇〇が動かない。修正して」と伝える。具体的な症状を書くのがコツ |
| ファイルが見つからない | 別のフォルダで作業している | エクスプローラーでフォルダ名を確認する。`bob-lab` と `lab-scenario` を間違えていないか確認する |
| `@` を入力してもメニューが出ない | 全角 `＠` を入力している、またはカーソルがチャット欄にない | 半角 `@` で入力し直す。チャット欄をクリックしてからやり直す |
| HTMLがブラウザーで開けない | ファイルの関連付けが設定されていない | 「エクスプローラーで表示する」→ ファイルをブラウザーにドラッグ＆ドロップする方法を案内する |
| 参加者間のスキル差が大きい | IT経験者と非経験者が混在 | 経験者には「追加機能の実装」や「他のチームのサポート」を依頼する。非経験者にはプロンプトのコピペで進められることを伝え、安心させる |

## 5. カスタマイズのヒント

### 業界別のアレンジ

参加者の業界が決まっている場合は、以下を事前に調整する。

- **課題カード**: 業界に合わせてペルソナや業務課題を書き換える
- **企業設定メモ**: 業界に合ったサンプル企業を用意しておく（例: 金融業なら「デジタル決済サービス PayQuick」）
- **サンプル・データ**: Bobに渡すプロンプトに業界特有のデータ例を含める

### 時間の伸縮

#### 1時間版（練習ラボのみ）

| パート | 所要時間 |
|---|---|
| 導入 | 3分 |
| Lab 1〜6 | 42分 |
| 練習ラボまとめ | 5分 |
| 振り返り | 10分 |

チーム演習を省略し、練習ラボの振り返りに時間を使う。

#### 2時間版（チーム演習を充実）

| パート | 所要時間 |
|---|---|
| 導入 | 5分 |
| Lab 1〜6 | 45分 |
| 練習ラボまとめ | 5分 |
| 企業設定 | 10分 |
| チーム演習 | 40分 |
| 成果物共有 | 10分 |
| 最終振り返り | 5分 |

チーム演習の時間を40分に拡大し、成果物共有を独立したパートにする。

#### 半日版（3〜4時間）

練習ラボ＋チーム演習に加え、以下を追加。

- オプション課題（`/plan` を使った計画駆動の開発）を全員で実施
- チーム演習後に、チーム間のクロス・レビュー（他チームのモックアップを操作してフィードバック）の時間を設ける
- 最後に「実務でどう活かすか」のディスカッション時間を設ける
