# Bob-a-thon ハンズオン・ガイド
仕様駆動のAI開発入門 — IBM Bobで学ぶモックアップ開発と文書化

* 最終更新日: 2026/4/13
* Author: Akira Onishi (Principal BDE, AI Lab Japan Office, IBM Japan)

---

## 📋 ワークショップ概要

### 目的

- **IBM Bobを使ったモックアップ開発を体験する**: 短時間でどこまで作れるか、その速さを実感する
- **「ざっくり依頼」と「仕様を決めてから依頼」の違いを知る**: 同じツールでも、渡し方で出来上がりが変わることを確かめる
- **要件 → モックアップ → 文書化の流れをひと通りやってみる**: 作って終わりではなく、要件定義や設計書も一緒に整える
- **ギーク思考を体感する**: スピード・オーナーシップ・サイエンス・オープンネスの4つの規範を、手を動かしながら実践する

### 対象者

事業担当者（LOB）、ITエンジニア、コンサルタントなど、どなたでも参加できる。
ITの知識は問わない。IBM Bobがパートナーとしてサポートする。

### 所要時間

約1時間30分（練習ラボ約45分 ＋ チーム演習・振り返り約45分）

### 前提条件

- IBM Bobが使える環境が準備されていること
- Webブラウザー（Chrome、Edge、Firefoxなど）が使えること
- 講師から配布された[ZIPファイル](https://github.com/IBM/japan-technology/raw/refs/heads/main/ibm-bob/bob-a-thon/new-business/lob/lab-files.zip)をダウンロードして、ローカル・コンピューター内に展開済みであること（[付録A: 配布フォルダの内容](#-付録a-配布フォルダの内容)を参照）

### The Geek Way（ギーク思考）について

参考書籍: [ギーク思考 — 圧倒的な結果を出す型破りな思考法](https://bookplus.nikkei.com/atcl/catalog/25/03/25/01931/)

このワークショップでは、ギーク思考の4つの規範を、手を動かしながら体感します。各Labに対応する規範を示しているので、「普段の仕事のやり方とどう違うか？」を意識しながら取り組んでみてください。

| 規範 | 考え方 | ワークショップでの実践 |
|---|---|---|
| 🏃 **スピード** | 綿密な計画よりも、素早く試して結果から学ぶ | まずやってみる。ダメなら別のやり方を試す。Bobとのやりとりを何度も繰り返す |
| 🙋 **オーナーシップ** | 共通のゴールに向かって、自分で考えて動く | 一人ひとりがリーダーであり実務家。自分の業務イメージをもってモックアップを作る |
| 🔬 **サイエンス** | 試した結果をもとに判断する。勘頼みにしない | やってみた結果から改善する。要件を整理し、根拠をもってモックアップを作る |
| 🌐 **オープンネス** | 情報を出し惜しみせず共有し、方向転換も歓迎する | 作ったものを見せ合う。ドキュメントを残して、他の人も使えるようにする |

## 🧪 練習ラボ（全員共通）

全員で同じ「営業チームのCRM」を題材に、IBM Bobを使った仕様駆動の開発プロセスを練習する。

> 💡 各Labの所要時間にはBobの応答待ち時間を含みます。応答時間は環境によって異なります。

### Lab 1: IBM Bobにざっくり依頼してみる（5分）

> 🏃 **ギーク思考: スピード** — まずやってみる。結果を見てから次を考える。

**このLabでやること**: 具体的な指示を出さずにBobに頼むと、どんなものが出てくるか試す。

#### 手順

**1-1. IBM Bobを起動する**

IBM Bobを開く。まだ起動していない場合は、アプリケーション一覧からIBM Bobを起動する。

**1-2. 作業用フォルダを作って開く**

1. IBM Bobのメニューから **「ファイル」→「フォルダーを開く」** を選ぶ
2. 任意の場所に **`bob-lab`** という名前でフォルダを新しく作る（Windowsの場合はドキュメント・フォルダー内、macOSの場合は書類フォルダー内がおすすめ）
3. 作った `bob-lab` フォルダを選んで **「開く」** をクリックする

> 💡 IBM Bobはフォルダ単位で作業します。Bobが作るファイルは、開いたフォルダの中に入ります。

**1-3. Bobにざっくり頼んでみる**

IBM Bobのチャット欄に、以下のメッセージを入力して送信する。

```
顧客関係管理（CRM）システムのモックアップを作って
```

Bobがファイルを作り始める。しばらく待つ（環境によって数十秒〜数分かかる）。

**1-4. 出来上がりを確認する**

1. ファイル一覧（エクスプローラー）に、Bobが作ったファイルが並ぶ
2. HTMLファイル（例: `index.html`）をクリックして中身を眺める
3. HTMLファイルを右クリック → **「エクスプローラーで表示する」**（macOSは **「Finderで表示する」**）を選ぶ
4. 開いたフォルダの中のHTMLファイルをダブルクリックすると、ブラウザーでモックアップが表示される

> 📝 **ここで振り返り**: Bobはそれなりのものを作ってくれたはずです。でも、自分が思い描いていた業務の流れや画面と比べて、どうでしょう？ 「ここが違う」「これが足りない」と感じたポイントを覚えておいてください。次のLabで、その差を埋める方法を学びます。

### Lab 2: 要件テンプレートを作る（5分）

Lab 1では「ざっくり頼む」を試しました。ここからは、もう少し丁寧にBobへ依頼するやり方を学んでいきます。

> 🔬 **ギーク思考: サイエンス** — 勘や好みだけで進めず、整理のための枠組みを用意する。

**このLabでやること**: 「モックアップの要件をどう整理すればいいか」をBobに相談して、事業担当者でも読み書きできるテンプレートを作ってもらう。

#### 手順

**2-1. Bobにテンプレート作成を頼む**

引き続き `bob-lab` フォルダで作業する。チャット欄に以下を貼り付けて送信してください。

> ⚠️ 長い文章ですが、そのまま全文コピーして貼り付けてください。

```
/ask ブラウザーでHTMLファイルを開くだけで動くモックアップの要件を整理するためのテンプレートを、mockup-req-template.md として作ってほしい。以下の方針で構成して。
- 事業担当者が読み書きできるように、アプリの目的・画面構成・機能一覧などの主要な要件を文書の前半にまとめる
- ストレージやセキュリティーなど技術的に難しい項目は、文書の後半にオプション扱いで記載する
- 技術選定の項目では、Webアプリ開発が初めての人でも迷わないよう、実績のある選択肢を既定値にする
- フォントの既定値は IBM Plex Sans JP にする
```

> 💡 **`/ask` コマンドとは**: BobのAskモードに切り替えるコマンドです。コードは書かず、調査やドキュメントの作成を頼むときに使います。詳しくは [IBM Bob IDEドキュメント](https://bob.ibm.com/docs/ide) を参照してください。

**2-2. できたテンプレートを見てみる**

1. ファイル一覧に `mockup-req-template.md` が出てくる。クリックして開く
2. 中身をざっと見て、以下の構成になっているか確認する：
   - **前半**: アプリの目的、画面構成、機能一覧など — 事業担当者が読み書きしやすい項目
   - **後半**: ストレージ、セキュリティーなど — オプション扱いの技術項目
   - **技術選定**: 初心者でも迷わない既定値が入っている
   - **フォント**: IBM Plex Sans JP が既定値になっている

> 📝 **ここがポイント**: このテンプレートを、次のLabでCRMの要件を書くときの「型」として使います。もし項目に過不足を感じたら、Bobに「〇〇の項目を追加して」と頼めば修正できます。このテンプレートは以降のLabでも参照するので、開いたままにしておいてください。

### Lab 3: 課題を読み解いて、要件をまとめる（10分）

Lab 2でテンプレート（要件の「型」）を作りました。このLabでは、実際の課題をもとに、その型に沿って要件をまとめます。

> 🔬 **ギーク思考: サイエンス** — 課題カードの情報をもとに、根拠のある要件を整理する。「なんとなく」ではなく「こういう理由でこう決めた」が言えるようにする。

**このLabでやること**: 配布フォルダにある営業チームの課題カードを読み、Lab 2のテンプレートの形式に沿ってCRMの要件をまとめる。ここからが「仕様駆動」のスタート。

#### IBM Bobでマークダウン・ファイルを新しく作る方法 (ここは、説明のみなので、実行しないでください）

このLabと以降のLabでは、マークダウン・ファイル（`.md`）を新しく作る場面がある。方法は2つある。


**方法1: エクスプローラーから作る（かんたん）**

1. ファイル一覧（エクスプローラー）の上部にある **「新しいファイル」アイコン**（紙に＋マーク）をクリックする
2. ファイル名を入力する（例: `crm_req.md`）
3. Enterキーを押すと、空のファイルが作られてエディタに表示される

**方法2: Bobのチャットから作る**

チャット欄で、ファイル名を指定して作成を頼む。例:

```
/code crm_req.md という空のマークダウン・ファイルを作って
```

> 💡 このハンズオンでは、Bobへの依頼の中でファイル名を指定するので、多くの場合Bobが自動的にファイルを作ってくれます。ファイルを自分で先に作っておきたい場合は、方法1を使うと確実です。

#### 手順

**3-1. 新しいウィンドウで配布フォルダを開く**

1. IBM Bobのメニューから **「ファイル」→「新しいウィンドウ」** を選ぶ
2. 新しいウィンドウで **「ファイル」→「フォルダーを開く」** を選ぶ
3. 事前に展開した配布フォルダ **`lab-scenario`** を選んで開く

* これ以降の作業は、このlab-scenarioフォルダー内で実行してください。

> 💡 `bob-lab` のウィンドウはそのまま残しておいてください。Lab 2のテンプレートを見比べたいときに便利です。

**3-2. フォルダの中身を確認する**

配布フォルダには以下のファイルが入っている（[付録A](#-付録a-配布フォルダの内容)も参照）。

| ファイル名 | 内容 | 使うタイミング |
|---|---|---|
| `company_profile.md` | 企業設定メモ（空欄） | チーム演習で使う（今は使わない） |
| `sales_challenge.md` | 営業チームの課題カード | **この練習ラボで使う** |
| `mockup-req-template-sample.md` | 要件テンプレートのサンプル | Lab 2で作ったものの参考例 |

**3-3. 営業チームの課題カードを読む**

`sales_challenge.md` を開いて、中身をひと通り読む。ポイントは以下の3点：

- **誰が使うか**（ペルソナ）: 営業担当者・田中さん（35歳、1日3〜5件の顧客対応）
- **何に困っているか**: 顧客情報の確認に手間取る、最適な提案ができない、対応履歴が残らない
- **最低限必要な機能**: 顧客リスト、顧客詳細画面、おすすめ商品の表示

**3-4. 要件をまとめる**

課題カードの内容を踏まえて、CRMの要件ファイルを作る。チャット欄に以下を入力して送信：

```
@sales_challenge.md の営業チームの課題と、@mockup-req-template-sample.md のテンプレート形式を参考にして、CRMモックアップの要件を crm_req.md にまとめて。以下の前提で書いて。
- ブラウザーでHTMLファイルを開くだけで動くモックアップとして作る（Webサーバーは使わない）
- 画面はPC・スマホ両対応のレスポンシブ・デザインにする
- フォントは IBM Plex Sans JP を使う
- サンプルの顧客データを5件以上含める
```

> 💡 **`@ファイル名` とは**: チャット欄で `@` を入力すると、メニューが表示されます。その中の **[Add File]** を選んでファイルを指定すると、そのファイルの内容をBobに読ませることができます（コンテキスト・メンション）。Bobがファイルの中身を理解したうえで回答してくれるので、的確な結果が得やすくなります。

**3-5. 出来上がった要件を確認する**

`crm_req.md` が作られる。開いて以下をチェック：

- 課題カードの3つの必須機能（顧客リスト、詳細画面、おすすめ表示）が入っているか
- 画面構成やデザインの方針が書かれているか
- 足りないと感じる項目があれば、Bobに「〇〇の項目を追加して」と頼んで修正する

### Lab 4: 要件からモックアップを作る（10分）

Lab 3で要件がまとまりました。いよいよ、その要件をBobに渡してモックアップを作ってもらいます。
* Lab 4の作業は、Lab 3と同じく、lab-scenarioフォルダー内で実行してください。

> 🏃 **ギーク思考: スピード** — 要件がまとまったら、すぐ形にする。完璧でなくていい、まず動くものを出す。

**このLabでやること**: Lab 3で作った要件ファイルを渡して、Bobにモックアップを作ってもらう。Lab 1の「ざっくり依頼」との違いを確かめる。

#### 手順

**4-1. 要件をもとにモックアップを作る**

チャット欄に以下を入力して送信する。

```
/code @crm_req.md の要件に従って、CRMモックアップを実装して。HTML・CSS・JavaScriptで構成し、最初に開くファイルは index.html にして。
```

> 💡 **`/code` コマンドとは**: BobのCodeモードに切り替えるコマンドです。ファイルの作成や修正など、実装作業を頼むときに使います。

Bobがモックアップを作る。しばらく待つ。

**4-2. ブラウザーで開いて確認する**

1. 生成された `index.html` を右クリックする
2. **「エクスプローラーで表示する」**（macOSは **「Finderで表示する」**）を選ぶ
3. 開いたフォルダの中の `index.html` をダブルクリックして、ブラウザーで表示する
4. 以下を見てみる：
   - 顧客リストは表示されているか
   - 顧客をクリックして詳細が見られるか
   - おすすめ商品/サービスは表示されているか

> 📝 **Lab 1と比べてみてください**: 要件を整理してから頼んだ今回と、ざっくり頼んだLab 1で、出来上がりにどんな違いがありますか？ 業務に必要な機能が入っているか、画面の構成は意図どおりか、比べてみましょう。

### Lab 5: 追加の要望でモックアップを磨く（7分）

Lab 4でモックアップの形ができました。ここからは、デザインや機能を追加しながら仕上げていきます。
* Lab 5の作業は、Lab 3, Lab 4と同じく、lab-scenarioフォルダー内で実行してください。
> 🏃 **ギーク思考: スピード** — 上手くいったらさらに良くする。小さな改善を素早く繰り返す。
> 🔬 **ギーク思考: サイエンス** — 変えた内容を要件定義に記録しておく。モックアップと仕様がズレないようにする。

**このLabでやること**: デザインや機能を追加するとき、モックアップだけでなく要件定義ファイルも一緒に更新する方法を練習する。

#### 手順

**5-1. デザイン変更を頼む**

チャット欄に以下を入力して送信する。

> 💡 `#1a1a2e` のような表記は「色コード」と呼ばれ、色を指定するための記法です。Bobはこの色コードを読み取って、指定した色をモックアップに反映してくれます。

```
/code 全体のデザインをダーク・テーマ（背景色: #1a1a2e、文字色: #e0e0e0、アクセント・カラー: #0f3460）に変更して。変更内容を crm_req.md のデザイン要件セクションにも反映して。
```

**5-2. 変わったか確かめる**

1. ブラウザーに戻り、ページを再読み込み（F5キーまたは更新ボタン）する。ダーク・テーマに変わったか確認する
2. IBM Bobで `crm_req.md` を開き、デザインの記述がダーク・テーマに書き換わっていることを確認する

> 📝 **ここがポイント**: 「crm_req.mdにも反映して」と付けるだけで、モックアップの見た目を変えると同時に、要件定義の記述も更新されます。こうしておけば、要件書とモックアップが食い違うことがありません。

**5-3. もう何回か試してみる（時間があれば）**

同じやり方で、好みに合わせてどんどん改善する。

例:

```
/code 画面左側に縦型のサイドバー・ナビゲーション（幅250px）を追加して、「顧客一覧」「ダッシュボード」「設定」のメニューを置いて。crm_req.md の画面構成セクションも更新して。
```

```
/code 顧客リストの各行に、ステータスに応じた色付きバッジ（新規: 青、商談中: 黄、成約: 緑、失注: 赤）を表示して。crm_req.md の機能要件セクションも更新して。
```

> 💡 毎回「crm_req.mdも更新して」と付けるのを忘れずに。これが仕様駆動のリズムです。

### Lab 6: 設計書と使い方説明書を作る（5分）

モックアップが仕上がったら、最後にドキュメントを揃えます。
* Lab 6の作業は、Lab 3, Lab 4, Lab 5と同じく、lab-scenarioフォルダー内で実行してください。

> 🌐 **ギーク思考: オープンネス** — 自分が作ったものを、他の人にも理解・利用してもらえる形にする。

**このLabでやること**: 出来上がったモックアップの設計書と使い方説明書を、Bobに書いてもらう。コードだけ作って終わりにせず、ドキュメントまで揃えるのが仕様駆動のやり方。

#### 手順

**6-1. 設計書と説明書を頼む**

チャット欄に以下を入力して送信する。

```
/code 以下の2つのマークダウン・ファイルを作って。
1. crm_design.md（設計書）: モックアップの画面構成、ファイル構成、使用技術、各画面の役割と遷移を記載
2. crm_readme.md（使い方説明書）: モックアップの開き方、各画面の操作方法、サンプル・データの説明を、初めて触る人向けに記載
```

**6-2. 出来上がりを確認する**

2つのファイルが作られる。それぞれ開いて、中身を見てみる。

| ファイル | 内容 |
|---|---|
| `crm_design.md` | 設計書 — 画面構成、ファイル構成、使っている技術、画面遷移などの設計情報 |
| `crm_readme.md` | 使い方説明書 — ブラウザーでの開き方、画面ごとの操作手順、サンプル・データの見方 |

### Bobへの頼み方のコツ

#### うまくいく頼み方

- 「営業担当者向けの顧客リスト画面を作って。名前・属性・ステータスをカード形式で並べて。」 → 誰向けか、何を表示するか、レイアウトが明確
- 「検索ボックスを上部に追加して、名前の部分一致で顧客リストを絞り込めるようにして。」 → 場所と動作が具体的
- 「ヘッダーの左に会社ロゴ、右にナビゲーション・メニュー（顧客一覧・設定）を追加して。」 → 配置と内容が明確

#### 避けた方がいい頼み方

- 「いい感じの画面を作って」 → 「いい感じ」はBobにはわからない
- 「データベースと連携して」 → モックアップの範囲を超えている
- 「完璧なシステムを作って」 → 範囲が広すぎて、何から手をつけるかBobが判断しにくい

### 段階的に作るのがコツ

1. **まず1画面**: シンプルな画面を1つ作る
2. **機能を足す**: 動くものをベースに、ひとつずつ機能を追加する
3. **見た目を整える**: 動作を確かめながら、デザインを調整する
4. **操作を追加**: クリックやフィルターなどのインタラクションを入れる

### 困ったときは

| 困りごと | やってみること |
|---|---|
| 思った通りに動かない | 何が違うかを具体的に伝える。例: 「検索ボックスに文字を入れてもリストが変わらない。入力した文字で顧客名を絞り込めるようにして。」 |
| デザインが好みと違う | 具体的にどう変えたいか伝える。例: 「ボタンをもっと大きくして、色を青(#1A73E8)にして。」 |
| 機能を追加したい | 一度にたくさん頼まず、ひとつずつ。まず基本を完成させてから次へ |
| ファイルが見当たらない | エクスプローラー（ファイル一覧）を確認。見つからなければ、Bobに「今あるファイルの一覧を教えて」と聞く |
| HTMLがブラウザーで開けない | HTMLファイルを右クリック →「エクスプローラーで表示する」→ ファイルをダブルクリック |
